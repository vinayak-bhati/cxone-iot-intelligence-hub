The Problem

Connected devices already know they are broken — but CXOne never hears it.

Silent Faults

Sensors detect the issue instantly — but that signal never reaches the contact center. The device knew before anyone else did.

Reactive-Only Support

Resolution starts only when the customer calls, waits on hold, and re-explains the problem from scratch. Hours or days are lost.

Blind, Costly Dispatch

Agents work with no device context and technicians roll out on guesswork — long AHT, repeat contacts, failed first visits.

Route IoT / device fault events natively into CXOne AI Agents to detect, triage and resolve service events autonomously — or escalate to a human with full device context — before the customer ever needs to call.

Under the Hood

How the Demo Works

For this Sparkathon prototype, three services simulate the flow above so the full journey can be demoed end-to-end today — the architecture is real, the wiring is a lightweight stand-in.

The real new component
event-normalizer

Genuinely the real component we'd ship — receives a simulated device fault and maps it to NICE's event schema. Validates the vendor payload against a JSON schema and publishes to AEP's ingest endpoint.

Mock stand-in
aep-stub

A small local service that mirrors NICE's real Agentic Engagement Plane contracts (Thread creation, Engagement Path routing, Handover, Context API) closely enough to prove the pattern.

Mock stand-in
cognigy-triage

Simulates the reasoning Cognigy would do — checking warranty and severity — using the same decision logic a real Cognigy Flow would run.

In production, these three steps run on NICE's actual AEP and Cognigy platforms — nothing here reinvents them. This mock exists only to prove the integration pattern live, in a hackathon timeframe.

Get Started

Run the Demo

Everything runs locally with a single double-click. Estimated cold-start time: under 2 minutes.

1
Clone the repository
git clone https://github.com/vinayak-bhati/cxone-iot-intelligence-hub.git
cd cxone-iot-intelligence-hub
2
Prerequisites

Node.js 18+ and npm (bundled with Node.js) must be installed. Internet access is required on first run only (npm install fetches packages; subsequent runs work offline).

Download from nodejs.org — the LTS installer bundles npm.

3
Start everything
Windows 
start-demo.bat    ← double-click, or right-click → Run as Administrator
Mac / Linux 
chmod +x start-demo.sh stop-demo.sh    # once only
./start-demo.sh

Four service windows open. The script installs dependencies on first run (30–60 seconds), then health-checks each service and opens your browser automatically.

4
Your browser opens automatically
http://localhost:5173/demo.html

If it doesn't open, paste that URL into your browser manually.

5
Try both scenarios (each takes under 30 seconds)

🟢 Fire Scheduled Maintenance — fires a LOW-severity scheduled maintenance event (device AC-00421, warranty ACTIVE). Watch the 5-step flow light up; Cognigy decides auto-resolve. A mock customer SMS appears. Proves the happy path.

🔴 Fire Critical Fault — fires a CRITICAL-severity compressor failure (device AC-00888, warranty ACTIVE). Flow lights up in red; Cognigy escalates to agent. Agent sees full device context, confirms warranty ACTIVE, books a repair technician and a customer SMS notification is sent. Proves the escalation + resolution path.

6
Stop everything
Windows
stop-demo.bat
Mac / Linux
./stop-demo.sh

Kills all processes on ports 3001 / 3002 / 3003 / 5173.

Troubleshooting — see README.md for the most current version

"Node.js not found" on startup
Install Node.js 18+ from nodejs.org and re-run the script. The LTS installer bundles npm automatically.
Port already in use (EADDRINUSE :3001 / :3002 / :3003 / :5173)
Run stop-demo.bat (Windows) or ./stop-demo.sh (Mac/Linux) first to kill stale processes, then restart.
Browser didn't open automatically
Paste http://localhost:5173/demo.html into your browser manually. The demo runs fine regardless of auto-open.
Timeline shows "Cannot reach Event Normalizer"
The services haven't finished starting yet — wait 10 seconds and try again. Check the service windows for any error output.
npm install takes a very long time
First-run only (30–60 seconds on a normal connection). Subsequent runs skip this step entirely. If it hangs, check your internet connection.
npx serve hangs or shows an error
npx downloads the serve package on first use — requires internet. Run npx serve web-ui -p 5173 manually from the submission folder, then open http://localhost:5173/demo.html.
